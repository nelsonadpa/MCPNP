## Implementation Steps

### Step 1: Create `_types.ts` — ServiceConfig interface
- [ ] `ServiceConfig` with: slug, name, serviceId, bitacoraBlock, hiddenFields, firstVisibleBlock, dropdownFields (key + searchTerms[]), textFields, dataGrid, fundamentacion, contactFields, checkboxKeys, bots[], status (pending → discovered → configured → passing → verified)
- [ ] Separate interfaces: `DropdownField`, `InputField`, `DataGridConfig`, `BotInfo`

### Step 2: Create `_seed-data.ts` — pre-populate from SERVICES-MAP
- [ ] All 18 services with: slug, name, serviceId, bitacoraBlock, buttonRef, NIT/Empresa keys
- [ ] Source: `countries/cuba/knowledge/SERVICES-MAP.md`

### Step 3: Extract `pe.config.ts` — reference config
- [ ] Extract PE's hardcoded constants from `pe-e2e-nuevo.spec.ts` into ServiceConfig format
- [ ] Includes: all hidden fields, dropdown fields with search terms, DataGrid config, bots
- [ ] Status: `verified`

### Step 4: Write `discovery.spec.ts` — automated form inventory
- [ ] Input: `SERVICE=fito` env var → looks up seed data
- [ ] Flow: navigate → company select → load form → dump raw inventory → set hidden fields → dump visible inventory → detect block structure
- [ ] Output: `discovery-output/{slug}.discovery.json` with component keys, types, visibility, required flags, plus `hints` section
- [ ] Screenshots at each phase

### Step 5: Write `service-e2e.spec.ts` — config-driven E2E
- [ ] Input: `SERVICE=fito` env var → loads `service-configs/fito.config.ts`
- [ ] Same proven logic as `service-e2e-template.spec.ts` but reads ALL constants from config
- [ ] Zero hardcoded values — dropdownFields, textFields, dataGrid all come from config
- [ ] Includes diagnostics when submit fails

### Step 6: Create `_registry.ts` — barrel + utilities
- [ ] `getConfig(slug)`, `getByStatus(status)`, `registrySummary()`
- [ ] Summary function for quick pipeline status check across all 18 services

### Step 7: Write `generate-observer.ts` — auto-generate observer artifacts
- [ ] Input: reads config for a service
- [ ] Output: `generated/{slug}-queries.md`, `generated/{slug}-dashboard.md`
- [ ] Templates follow PE's `pe-dashboard.md` and `pe-queries.md` structure

### Step 8: Validate PE through new system
- [ ] Run `SERVICE=pe npx playwright test specs/service-e2e.spec.ts`
- [ ] Must pass identically to `pe-e2e-nuevo.spec.ts`

### Step 9: Add `discovery` project to `playwright.config.ts`
- [ ] Headless by default for batch runs
- [ ] Same auth state + base URL
- [ ] Add npm scripts to `package.json`

## Pipeline Per Service (post-implementation)

```
Phase 1: Discovery (~5 min, automated)
  SERVICE=fito npx playwright test specs/discovery.spec.ts --project=cuba-frontoffice

Phase 2: Config generation (~10 min, agent + human)
  Agent reads discovery JSON + seed data → drafts fito.config.ts
  Human reviews, adjusts dropdown search terms

Phase 3: E2E run (~5 min, automated)
  SERVICE=fito npx playwright test specs/service-e2e.spec.ts --project=cuba-frontoffice

Phase 4: Debug (~10 min, 1-2 iterations)
  Fix failed dropdowns/fields in config → re-run

Phase 5: Observer (~5 min, Observer agent)
  npx ts-node scripts/generate-observer.ts fito
  Observer runs Graylog queries → fills real numbers → verified
```

**~35 min per service** vs current ~2.5h

## Agent Roles

|-------|-------|--------|

| Discovery | Tester | Run `discovery.spec.ts` per service |

| E2E execution | Tester | Run `service-e2e.spec.ts`, debug failures |
| Observer validation | Observer | Run Graylog queries, fill dashboards |
| Knowledge update | Tester | Update SERVICES-MAP E2E column, update registry status |
